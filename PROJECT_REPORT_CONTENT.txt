ABSTRACT (Page 5)
================================================================================

Traditional attendance systems — physical punch cards, biometric devices, and
manual registers — have long suffered from fundamental flaws: buddy punching, ghost
employees, inability to support remote or multi-site workforces, and complete lack
of real-time visibility. As organisations grow more distributed, these limitations
translate directly into payroll fraud, compliance failures, and operational
inefficiency.

This report presents the design and implementation of a production-grade,
Geo-Location Based Attendance System that addresses all of these problems through
a modern, mobile-first technology stack. The system enables employees to check in
and check out of work using their smartphones — GPS coordinates are validated
against administrator-defined geofences (either circular radius zones or arbitrary
polygon boundaries) before any attendance record is accepted.

The platform is built as a microservices-oriented architecture: a FastAPI (Python
3.12) REST and WebSocket backend, a PostgreSQL 16 + PostGIS database for spatial
queries, Redis 7 for caching and real-time pub/sub, Celery for background task
processing, MinIO for object storage, and a React Native (Expo 51) mobile
application that runs natively on both Android and iOS.

Key innovations include a multi-layered fraud detection engine that combines VPN/
proxy detection (IPQualityScore API), mock GPS detection, impossible travel speed
checks using the Haversine formula, and device fingerprinting — each contributing
to a composite fraud score (0.0–1.0) stored with every attendance record. A manual
approval workflow with real-time WebSocket push notifications, configurable SLA
escalation via Celery Beat, and a full audit trail complements the automated
pipeline. Admin dashboards provide live heatmaps, anomaly feeds, trend charts, and
exportable CSV/PDF reports.

The result is a scalable, secure, and extensible platform capable of supporting
multi-organisation, multi-site deployments with role-based access control, streak-
based gamification for employee engagement, and Prometheus-powered observability.

Keywords: Geo-fencing, Attendance Management, GPS Check-in, Fraud Detection,
FastAPI, React Native, PostGIS, Real-Time Analytics, Mobile-First, Multi-tenant.

1.1 BACKGROUND OF THE WORK
----------------------------

Attendance management is one of the most fundamental administrative functions in
any organisation. Accurate tracking of when employees start and end work directly
impacts payroll accuracy, regulatory compliance, workforce planning, and overall
organisational productivity. Despite this importance, the majority of organisations
— particularly small-to-medium enterprises, construction firms, healthcare
facilities, logistics companies, and educational institutions — still rely on
outdated mechanisms that introduce significant risks.

THE PROBLEM WITH EXISTING APPROACHES

Manual registers and paper-based systems are error-prone, easily falsified, and
provide no real-time data. A supervisor reviewing paper timesheets at the end of
the week has no visibility into attendance patterns as they unfold.

Biometric systems (fingerprint scanners, iris readers) address forgery but
introduce their own problems: high hardware cost, maintenance overhead, hygiene
concerns (especially post-COVID), complete inability to support remote workers, and
single-point-of-failure architecture where one broken device halts the entire
process for a site.

RFID/NFC card-based systems are cheap but trivially defeated by "buddy punching" —
one employee clocking in on behalf of an absent colleague. There is also no GPS
correlation, meaning the system cannot verify whether the employee is actually at
the correct location.

Basic GPS mobile apps without fraud detection are vulnerable to mock GPS attacks
(GPS spoofing using apps that broadcast fake coordinates), VPN tunnelling that masks
the true IP origin of a check-in request, and impossible travel (an employee
clocking in at two geographically distant sites within minutes).

THE OPPORTUNITY

Consumer smartphone penetration now exceeds 85% in most urban and semi-urban
markets. Every modern smartphone ships with a high-accuracy GPS receiver, a cellular
or Wi-Fi internet connection, and a capable processor. This creates an opportunity
to build an attendance system that:

- Requires no additional hardware beyond devices employees already carry
- Validates physical presence at the correct location in real time
- Detects and flags fraudulent check-in attempts automatically
- Provides live dashboards that transform attendance data into actionable insights
- Scales horizontally from a single-site company to a global enterprise

This project realises that opportunity with a production-grade, containerised
platform built entirely on open-source technology.

SCOPE OF THE SYSTEM

The system covers the full lifecycle of employee attendance:
- Organisation and site setup (multi-tenant, multi-site)
- Geofence definition (circular radius or polygon boundary per site)
- Employee onboarding with role assignment
- Shift template creation and user assignment
- Real-time GPS check-in and check-out with fraud scoring
- Break tracking within shifts
- Manual approval workflows for edge cases (e.g., GPS unavailable)
- Admin analytics with heatmaps, trend charts, and CSV/PDF export
- Celery-driven background jobs for SLA escalation, report generation, and cleanup
- Real-time notifications via WebSocket and Firebase Cloud Messaging (FCM)
- Prometheus metrics for operational observability

The system is intentionally scoped to exclude payroll computation (which varies
widely by jurisdiction and organisation) and leave that integration to external
HRMS systems via the provided REST API.

CHAPTER 2 — LITERATURE REVIEW (Page 14)
================================================================================

TRADITIONAL ATTENDANCE SYSTEMS

Early research by Adewoye et al. (2015) and Kumar & Singh (2017) documented
widespread payroll fraud caused by buddy punching in manual and card-based systems,
with estimates suggesting 2–5% of gross payroll is lost to time theft in large
organisations. Biometric systems were proposed as the solution throughout the 2010s
but adoption was limited by hardware cost and maintenance complexity.

GPS-BASED ATTENDANCE — PRIOR WORK

Multiple studies explored GPS-based check-in starting in 2013. Al-Mouh et al. (2015)
built a simple Android application that sent GPS coordinates to a server and compared
them against a static list of allowed locations. The system lacked fraud detection,
real-time notifications, or any analytics capability. Kalyan et al. (2019) added a
geofencing boundary but still used a monolithic PHP backend with polling-based
status updates rather than real-time push.

The core gap identified across all prior work: GPS coordinates alone are trivially
spoofed using widely available mock GPS applications (e.g., "Fake GPS Location"
with over 10 million downloads on the Google Play Store). None of the surveyed
systems implemented any mechanism to detect or reject spoofed coordinates.

GEOFENCING TECHNOLOGY

Geofencing using the Haversine formula for circular boundaries is well-established.
More recent work by Deng et al. (2020) demonstrated the use of Shapely (a Python
computational geometry library) for arbitrary polygon geofences, enabling irregular
boundary shapes such as campus buildings, factory floors, or L-shaped compounds.
This project implements both approaches, selecting the polygon check when a polygon
is defined and falling back to radius check otherwise.

FRAUD DETECTION IN LOCATION SERVICES

IP-based fraud detection (VPN/proxy/Tor detection) is commercially mature — services
like IPQualityScore, MaxMind, and IP-API offer real-time classification of IP
addresses. Integrating these APIs at check-in time allows the system to flag check-
ins submitted through anonymising proxies.

Impossible travel detection was popularised by financial fraud systems (credit card
companies have used it since the 1990s). Applied to attendance, if an employee
checks in at Site A at 08:00 and then checks in at Site B at 08:15, and the
Haversine distance between the sites divided by the time interval yields a speed
above a threshold (e.g., 900 km/h, ruling out ground transport), the system flags
the second record as potentially fraudulent.

REAL-TIME NOTIFICATION ARCHITECTURES

WebSocket-based real-time updates (RFC 6455) have become the industry standard for
push notifications in web and mobile applications. Redis pub/sub provides an
effective message broker to fan out events to multiple connected WebSocket clients
in a horizontally-scaled backend. This is the approach adopted in this project,
supplemented by Firebase Cloud Messaging (FCM) for push notifications to mobile
devices that are not actively connected.

COMPETING PRODUCTS

Commercial solutions in this space include Darwinbox, greytHR, ADP, and Keka (for
Indian market), and TimeClock Plus, Replicon, and Workday for global markets.
These products offer GPS attendance but typically: (a) charge per-seat SaaS fees
that are prohibitive for smaller organisations, (b) are closed-source with no
customisation capability, (c) bundle attendance with payroll as a forced package,
and (d) do not expose a developer API for integration with custom systems.

This project fills the gap by providing a self-hostable, open-architecture platform
where the full source code is available, all features are accessible via a RESTful
API, and organisations can deploy it on their own infrastructure.

TECHNOLOGY SELECTION RATIONALE

FastAPI was chosen over Flask or Django REST Framework for its native async support
(critical for handling concurrent WebSocket connections and external API calls),
automatic OpenAPI documentation generation, and Pydantic-based schema validation.
React Native with Expo was selected over Flutter or native Android/iOS development
for its shared codebase, rapid iteration capability, and excellent location/maps
library ecosystem. PostgreSQL + PostGIS was chosen over MongoDB or MySQL for its
first-class spatial indexing support (GiST indexes on geometry columns), ACID
compliance, and mature async driver (asyncpg via SQLAlchemy).

3.1 PROJECT OBJECTIVES
-----------------------

PRIMARY OBJECTIVES:
1. Design and implement a mobile-first attendance system that validates employee
   presence at authorised locations using real-time GPS geofencing.
2. Build a multi-layered fraud detection pipeline that identifies VPN usage, GPS
   spoofing, impossible travel, and device anomalies.
3. Provide a real-time analytics dashboard for administrators with heatmaps, KPIs,
   and trend visualisations.
4. Implement a manual approval workflow with real-time WebSocket notifications and
   configurable SLA escalation.
5. Package the entire system as a containerised deployment using Docker Compose for
   easy self-hosting.

SECONDARY OBJECTIVES:
6. Support multi-organisation and multi-site deployments from a single instance.
7. Implement role-based access control (RBAC) with four distinct roles.
8. Provide a gamification layer (streak counting) to incentivise consistent attendance.
9. Expose Prometheus metrics for production observability.
10. Generate exportable CSV and PDF attendance reports as background jobs.

3.2 PRODUCT REQUIREMENTS (PRD)
--------------------------------

FUNCTIONAL REQUIREMENTS

FR-01: Authentication
  - Employees and admins authenticate using email + password credentials.
  - JWT access tokens with configurable expiry are issued on login.
  - Refresh token rotation is supported.
  - Role is encoded in the JWT payload and validated on every protected endpoint.

FR-02: Organisation & Site Management
  - Super admin can create organisations.
  - Org admin can create and manage work sites.
  - Each site has a name, address, optional center coordinates and radius (metres),
    and an optional polygon boundary (GeoJSON format).
  - Sites can be activated or deactivated without deleting historical data.

FR-03: Geofence Check-In / Check-Out
  - Employee mobile app collects GPS coordinates (latitude, longitude, accuracy).
  - On check-in, the server evaluates whether the coordinate lies within the site's
    geofence — using polygon check if defined, otherwise radius check.
  - If outside geofence, check-in is rejected with a descriptive error message.
  - Check-out must be paired with an open check-in; unpaired check-out is rejected.
  - Break start and break end are tracked as separate event types within a shift.

FR-04: Fraud Detection
  - Every check-in event triggers a fraud evaluation pipeline.
  - Fraud score (0.0–1.0) and individual fraud flags are stored with every record.
  - The pipeline checks: VPN/proxy/Tor (via IPQualityScore API with Redis-cached
    responses, TTL 1 hour; falls back to ip-api.com), mock GPS client flag, device
    fingerprint consistency, and impossible travel speed.
  - Records exceeding the org's configured fraud threshold are marked is_valid=False.
  - Administrators can review flagged records in the anomaly feed.

FR-05: Manual Approval Workflow
  - Employees can submit a manual attendance request (e.g., when working from a
    client site not in the system) including a reason text and optional photo.
  - Photos are stored in MinIO object storage; a pre-signed URL is returned.
  - The request is pushed in real time to the assigned supervisor via WebSocket.
  - If not actioned within the SLA period (configurable, default 2 hours), Celery
    Beat triggers an escalation notification to the org admin.
  - Approve/reject decision is recorded with the reviewer's ID and timestamp.
  - On approval, an attendance record is created with is_manual=True.

FR-06: Shift Management
  - Org admin can define shift templates: name, start_time, end_time, days_of_week,
    late_threshold_minutes.
  - Shifts are assigned to users with effective_from / effective_to date ranges.
  - A shift with effective_to=NULL represents an open-ended ongoing assignment.

FR-07: Analytics and Reporting
  - Admin dashboard exposes: total employees, present today, absent today, late today,
    average attendance rate (30-day rolling), and top anomalies.
  - Heatmap endpoint returns aggregated presence counts per site per hour.
  - Trend chart endpoint returns daily attendance totals over a configurable period.
  - CSV and PDF report generation is handled as a Celery background job; the download
    URL (MinIO pre-signed) is returned to the client on completion.

FR-08: Real-Time Feed
  - WebSocket endpoint /ws/feed broadcasts every check-in and check-out event to
    all connected admin clients in real time via Redis pub/sub.
  - WebSocket endpoint /ws/approvals broadcasts new approval requests and decisions.
  - JWT is passed as a query parameter (?token=...) for WebSocket authentication.

FR-09: Notifications
  - Email notifications are sent via SMTP (MailHog in development).
  - Push notifications are sent to mobile devices via Firebase Cloud Messaging (FCM).
  - Notification preferences are stored per user.

FR-10: Audit Trail
  - Every state-changing action (check-in, approval decision, site update, user
    role change) is recorded in the audit_logs table with actor_id, action, resource
    type, resource ID, and timestamp.

NON-FUNCTIONAL REQUIREMENTS

NFR-01: Performance — API response time < 200ms at p95 for standard read endpoints.
NFR-02: Scalability — Backend is stateless; multiple uvicorn instances can run behind
         Nginx without shared session state (JWT-based auth, Redis for state).
NFR-03: Security — Passwords hashed with bcrypt (cost factor 12). JWT secrets
         configurable. CORS origins whitelisted. SQL injection prevented by ORM
         parameterisation. Rate limiting configured at Nginx level.
NFR-04: Availability — Health check endpoints (/health, /health/ready) expose
         liveness and readiness for orchestration platforms.
NFR-05: Observability — Prometheus metrics (request count, latency histograms) on
         /metrics endpoint. Structured logging via Python logging module.
NFR-06: Portability — Full Docker Compose deployment. Single .env file for all
         configuration. No cloud-provider-specific dependencies.

3.3 SYSTEM ARCHITECTURE OVERVIEW
----------------------------------

The system follows a layered, microservices-inspired architecture packaged within
a single Docker Compose file for ease of deployment:

LAYER 1 — CLIENT TIER
  React Native Expo 51 mobile app (iOS + Android + Web)
  Communicates via HTTPS REST and WSS WebSocket

LAYER 2 — GATEWAY TIER
  Nginx reverse proxy handles TLS termination, WebSocket upgrade headers,
  rate limiting (1000 req/min per IP), and static asset serving.

LAYER 3 — APPLICATION TIER
  FastAPI (Python 3.12) with Uvicorn ASGI server.
  Async request handling with SQLAlchemy async sessions.
  8 REST router modules + WebSocket module.
  Prometheus metrics middleware (request count, latency).

LAYER 4 — BACKGROUND PROCESSING TIER
  Celery workers process: approval SLA escalation, report generation (CSV/PDF),
  geofence boundary watch (alerting when employees leave a site mid-shift), and
  database cleanup (soft-deleting expired records).
  Celery Beat scheduler triggers periodic tasks.

LAYER 5 — DATA TIER
  PostgreSQL 16 + PostGIS 3.4 — primary relational store with spatial indexing.
  Redis 7 — response cache (geofence data, IP fraud scores), Celery broker/backend,
             WebSocket pub/sub channel.
  MinIO — S3-compatible object store for employee photos and generated reports.

3.4 METHODOLOGY
----------------

Development followed an iterative, phased approach:

PHASE 1 — Foundation (Database models, configuration, Docker Compose)
PHASE 2 — Authentication and User Management (JWT, RBAC, registration)
PHASE 3 — Core Attendance (Check-in/out, geofence evaluation, shift tracking)
PHASE 4 — Fraud Detection (VPN detection, mock GPS, impossible travel, scoring)
PHASE 5 — Approvals, Notifications, WebSocket (Real-time pipeline)
PHASE 6 — Analytics, Reporting, Background Jobs, Mobile App completion

Each phase was completed before the next began, with integration testing at each
boundary. The mobile app was developed in parallel from Phase 3 onward, consuming
the REST API as it was made available.

4.2 DATABASE IMPLEMENTATION
-----------------------------

The database schema comprises 9 core tables:

TABLE: organisations
  Stores multi-tenant organisation records. Each organisation has a unique UUID
  primary key, name, subscription tier (free/pro/enterprise), and timestamps.
  All user and site records cascade-delete when an organisation is removed.

TABLE: users
  Stores authenticated users. Key fields:
  - id (UUID, PK)
  - org_id (FK → organisations, CASCADE DELETE)
  - supervisor_id (FK → users self-referential, for supervisor hierarchy)
  - email (unique, indexed)
  - hashed_password (bcrypt)
  - role (ENUM: super_admin, org_admin, supervisor, employee)
  - streak_count, last_checkin_date (gamification)
  - is_active (soft-disable without deletion)

TABLE: sites
  Represents a physical work location.
  - center_lat, center_lng (Float, decimal degrees)
  - radius_meters (Float, configurable per site)
  - polygon (Text — GeoJSON stored as string, parsed by Shapely at runtime)
  - Polygon takes precedence over radius when defined.

TABLE: attendance_records
  The central fact table. Every check-in, check-out, break_start, and break_end is
  a separate row. Key fields:
  - event_type (ENUM: checkin, checkout, break_start, break_end)
  - lat, lng (Float, 10-digit precision)
  - accuracy_meters (GPS accuracy radius reported by device)
  - ip_address (String 45, IPv6-safe)
  - fraud_score (Float 0.0–1.0, composite)
  - fraud_flags (JSONB — individual check results, e.g. {"vpn_detected": true})
  - is_valid (Boolean — False if fraud_score > org threshold)
  - is_manual (Boolean — True for manual approval records)
  - photo_url (Text — MinIO URL for selfie, nullable)
  - FK to users, sites, shifts, devices, manual_approval_requests

TABLE: shifts
  Reusable shift templates. Fields:
  - name (String 100)
  - start_time, end_time (Time)
  - days_of_week (ARRAY[Integer] — ISO weekday numbers 1=Monday)
  - late_threshold_minutes (Integer, default 15)
  - is_active (Boolean)

TABLE: user_shifts
  Junction table assigning shifts to users with date ranges (effective_from,
  effective_to). effective_to=NULL means open-ended.

TABLE: manual_approval_requests
  Stores employee-submitted requests for manual attendance approval.
  - status (ENUM: pending, approved, rejected, escalated)
  - reason (Text)
  - photo_url (MinIO URL for evidence photo)
  - reviewed_by (FK → users), reviewed_at (timestamp)
  - sla_deadline (timestamp, used by Celery Beat escalation task)

TABLE: devices
  Device fingerprinting table. Stores device_id (UUID), device_fingerprint (String),
  platform (iOS/Android), and user_id. Anomalous device changes trigger fraud flags.

TABLE: audit_logs
  Immutable audit trail. Stores actor_id, action (String), resource_type, resource_id,
  metadata (JSONB), and created_at. Write-only — no updates or deletes.

ADDITIONAL: notifications table for per-user notification records.

DATABASE DESIGN DECISIONS:
- UUID primary keys used throughout (gen_random_uuid()) to prevent enumeration attacks
  and support distributed ID generation if sharding is introduced later.
- JSONB for fraud_flags allows flexible schema evolution — new fraud checks can be
  added without a schema migration.
- Alembic async env.py handles all schema migrations. The initial migration creates
  the PostGIS extension before any spatial operations.
- Indexes on: user_id in attendance_records (composite queries), created_at
  (time-range queries), email in users (login lookups).

4.3 FRONTEND (MOBILE APP) IMPLEMENTATION
-----------------------------------------

The mobile application is built with React Native Expo 51 using file-based routing
via Expo Router, mirroring the directory structure of Next.js:

  mobile/app/
  ├── (auth)/          # Unauthenticated screens (login)
  ├── (employee)/      # Employee stack (index, checkin, history, streaks, profile)
  └── (admin)/         # Admin stack (dashboard, geofences, approvals, employees,
                       #              reports, anomalies)

AUTHENTICATION FLOW:
  The root layout (_layout.tsx) checks the Zustand auth store on mount. If no token
  is present, the user is redirected to (auth)/login. On successful login, the JWT
  is persisted in SecureStore (encrypted device storage) and the user is navigated
  to the appropriate stack based on their role.

KEY SCREENS:

Employee Check-In Screen (mobile/app/(employee)/checkin.tsx):
  - Requests foreground location permission on first use.
  - Displays a Mapbox map with the user's current GPS position and the target
    geofence boundary rendered as an overlay.
  - "Check In" button sends lat/lng/accuracy to POST /api/v1/attendance/checkin.
  - Server response includes fraud_score and a human-readable status message.
  - If outside geofence, a descriptive error card is shown.
  - GPS accuracy indicator shows whether the reading is reliable (<10m = excellent,
    10–50m = good, >50m = poor — user advised to move outdoors).

Admin Dashboard Screen (mobile/app/(admin)/dashboard.tsx):
  - Displays KPI cards: Present Today, Absent Today, Late Today, Attendance Rate %.
  - LiveFeed component subscribes to /ws/feed and appends real-time check-in events
    to a scrollable list without polling.
  - HeatmapLayer renders aggregated employee presence data on the Mapbox map.
  - Quick navigation tiles link to Geofences, Approvals, Employees, Reports.

Geofence Editor Screen (mobile/app/(admin)/geofences.tsx):
  - MapGeofenceEditor component enables drawing polygon boundaries directly on the map.
  - Supports editing existing geofences — tap-to-add-vertex, drag-vertex, tap-to-
    remove-vertex.
  - Serialises the drawn polygon as GeoJSON before sending to PUT /api/v1/locations/{id}.

Approvals Screen (mobile/app/(admin)/approvals.tsx):
  - Subscribes to /ws/approvals for real-time incoming requests.
  - Each request card shows employee name, reason, photo thumbnail, and time.
  - Approve/Reject buttons call PATCH /api/v1/approvals/{id}.

Streaks Screen (mobile/app/(employee)/streaks.tsx):
  - Displays the employee's current check-in streak (consecutive days present).
  - StreakBadge component renders animated badge with streak milestone labels
    (7 days = "Week Warrior", 30 days = "Month Master", etc.).

STATE MANAGEMENT:
  Three Zustand stores are used:
  - authStore: JWT token, user profile, login/logout actions
  - attendanceStore: today's records, check-in state, streak data
  - notificationStore: unread notification count, notification list

CUSTOM HOOKS:
  - useLocation: wraps expo-location, returns current coords with accuracy
  - useGeofence: given a site_id, returns whether user is within geofence (client-side
    preview; server performs the authoritative check)
  - useWebSocket: manages WebSocket connection lifecycle with automatic reconnect on
    disconnect (exponential backoff: 1s, 2s, 4s, 8s, max 30s)

API CLIENT:
  Axios instance in mobile/services/api.ts configured with base URL from environment
  variable. Request interceptor attaches Authorization: Bearer {token} header from
  authStore. Response interceptor handles 401 Unauthorized by clearing auth state
  and redirecting to login screen.

4.4 BACKEND LOGIC IMPLEMENTATION
----------------------------------

FASTAPI APPLICATION STRUCTURE:

The FastAPI app is created by the create_app() factory in app/main.py. This factory:
1. Initialises PostGIS extension and creates tables via init_db() on startup.
2. Creates the MinIO bucket if absent.
3. Pings Redis to confirm connectivity.
4. Registers all 8 router modules under the /api/v1 prefix.
5. Attaches global exception handlers (422 validation, 404, 500).
6. Registers Prometheus middleware, CORS middleware.
7. Registers WebSocket router and health/metrics standalone routes.

KEY ROUTERS:

auth router (/api/v1/auth):
  POST /login — validates credentials, issues JWT with role claim
  POST /register — creates user with bcrypt-hashed password
  POST /refresh — rotates refresh token

attendance router (/api/v1/attendance):
  POST /checkin — validates geofence, runs fraud pipeline, records event
  POST /checkout — validates open check-in exists, records checkout
  GET /today — returns today's records for the authenticated user
  GET /history — paginated history with date range filter

GEOFENCE EVALUATION LOGIC (geo_service.py):
  Step 1: Load site from Redis cache (10-minute TTL) or database.
  Step 2: If site has a polygon (GeoJSON string), parse it and use Shapely's
          polygon.contains(Point(lng, lat)) for containment check.
          Note: GeoJSON uses (longitude, latitude) ordering, not (lat, lng).
  Step 3: If no polygon, use haversine_distance() to compute km between check-in
          point and site center; multiply by 1000 to compare against radius_meters.
  Step 4: Return boolean result to the attendance router.

FRAUD DETECTION PIPELINE (fraud_service.py):
  Triggered for every check-in event:

  Check 1 — VPN/Proxy/Tor Detection:
    Call IPQualityScore API with the client IP address. Response includes
    is_vpn, is_proxy, is_tor, fraud_score. Cache result in Redis with 1-hour TTL.
    If IPQualityScore unavailable, fall back to ip-api.com (proxy field).

  Check 2 — Mock GPS Detection:
    The mobile client includes a boolean flag is_mock_location in the request body.
    The app sets this based on Android's Location.isFromMockProvider() and iOS
    equivalent. This is a client-side signal; the server records but cannot
    independently verify it (server-side trust model treats this as advisory).

  Check 3 — Impossible Travel:
    Query the user's most recent check-in from the past 12 hours.
    If found, compute haversine distance between the previous location and current.
    Compute time elapsed in hours.
    If speed > 900 km/h (above any realistic ground transport), flag as impossible.

  Check 4 — Device Fingerprint:
    The mobile client sends a device_fingerprint hash computed from hardware
    identifiers (device model, OS version, app installation ID).
    If the fingerprint differs from the last registered device for this user,
    flag as device_change. Multiple rapid device changes within a short window
    trigger an elevated fraud score.

  Composite Scoring:
    fraud_score = weighted sum of individual check scores.
    Weights (configurable): VPN = 0.4, mock_gps = 0.35, impossible_travel = 0.5,
    device_change = 0.15. Score is clamped to [0.0, 1.0].
    If fraud_score > org.fraud_threshold (default 0.6), is_valid = False.

APPROVAL WORKFLOW (approval_service.py):
  Step 1: Employee creates ManualApprovalRequest via POST /api/v1/approvals.
          Photo (if provided) is uploaded to MinIO; pre-signed URL stored.
  Step 2: approval_service broadcasts the request via Redis pub/sub channel
          "approvals:{org_id}".
  Step 3: All admin/supervisor WebSocket clients subscribed to /ws/approvals
          receive the event in real time.
  Step 4: Celery Beat runs the escalation task every 15 minutes, checking for
          pending requests that have passed their sla_deadline. Escalated requests
          get status="escalated" and a notification is sent to org admin.
  Step 5: Reviewer calls PATCH /api/v1/approvals/{id} with decision=approved/rejected.
          On approval, an AttendanceRecord is created with is_manual=True.
          On both decisions, the result is broadcast via WebSocket and FCM push.

WEBSOCKET IMPLEMENTATION (websocket.py):
  Two WebSocket routes managed as an APIRouter:
  - /ws/feed: admin check-in feed
  - /ws/approvals: approval queue

  Each endpoint:
  1. Extracts JWT from ?token= query parameter.
  2. Validates token and fetches user.
  3. Subscribes to the appropriate Redis pub/sub channel.
  4. Enters an async receive loop, forwarding messages from Redis to the WebSocket
     client. If the client disconnects, the subscription is cancelled.

CELERY TASKS (workers/tasks/):
  - escalation.py: Escalates overdue approval requests
  - reports.py: Generates CSV and PDF attendance reports, uploads to MinIO
  - geofence_watch.py: Checks if employees who checked in have left the geofence
    (useful for multi-hour shifts in bounded areas)
  - cleanup.py: Soft-deletes or archives records older than configurable retention
    period; purges expired Redis keys

4.5 SECURITY AND ACCESS CONTROL
---------------------------------

AUTHENTICATION:
  JWT (JSON Web Tokens) are used for stateless authentication. On login, a signed
  JWT is issued with payload: {"sub": user_id, "role": user_role, "exp": expiry}.
  The secret key (JWT_SECRET) is loaded from environment variables — never hardcoded.
  Tokens have a configurable expiry (default: 60 minutes for access tokens).

ROLE-BASED ACCESS CONTROL (RBAC):
  Four roles with hierarchical permissions:
  - super_admin: Full platform access, can manage organisations
  - org_admin: Full access within their organisation (users, sites, analytics)
  - supervisor: Can approve/reject manual requests for their team; view team analytics
  - employee: Can check in/out, view own history, submit manual requests

  Dependency injection in FastAPI:
  - get_current_user() — any authenticated user
  - require_role(UserRole.org_admin) — restricts to specified role or above
  - require_same_org() — prevents cross-organisation data access

ADDITIONAL SECURITY MEASURES:
  - Passwords hashed with bcrypt, cost factor 12 (strong enough to resist brute force)
  - UUID primary keys prevent sequential enumeration of records
  - All database queries use SQLAlchemy ORM (parameterised) — no raw SQL string
    interpolation, preventing SQL injection
  - CORS origins are whitelisted (not wildcard in production)
  - Nginx enforces rate limiting: 1000 requests/minute per IP
  - Nginx enforces request size limits to prevent large-payload attacks
  - MinIO pre-signed URLs expire after configurable duration (default: 1 hour)
  - Sensitive fields (hashed_password) excluded from all Pydantic response schemas
  - Audit logs record all state-changing operations with actor identity

4.7 CHALLENGES AND SOLUTIONS
------------------------------

CHALLENGE 1 — GPS Accuracy Indoors
  GPS accuracy degrades significantly indoors (15–50m typical). If the geofence
  radius is set too tight (e.g., 50m), indoor employees near the building perimeter
  may be rejected despite being physically present.
  SOLUTION: The system stores accuracy_meters with every record and allows
  administrators to configure site radius with awareness of this constraint.
  A client-side accuracy indicator guides employees to move to better signal.
  Future: Wi-Fi positioning or iBeacon integration for indoor accuracy.

CHALLENGE 2 — WebSocket at Scale
  A single FastAPI/uvicorn process handles all WebSocket connections in memory.
  If multiple uvicorn instances run behind Nginx (for scaling), each connection
  is isolated to one process.
  SOLUTION: Redis pub/sub is used as the message fan-out layer. All uvicorn
  instances subscribe to the same Redis channels. A check-in event published by
  any instance is received by all WebSocket subscribers regardless of which
  instance they are connected to.

CHALLENGE 3 — Celery Sync vs FastAPI Async
  FastAPI uses async SQLAlchemy (asyncpg driver). Celery runs in a synchronous
  worker environment that cannot use async sessions.
  SOLUTION: Celery tasks use a separate synchronous SQLAlchemy engine with the
  psycopg2 driver, entirely independent from the FastAPI async engine.

CHALLENGE 4 — Geofence Polygon GeoJSON Ordering
  GeoJSON specifies coordinates as [longitude, latitude] (X, Y), but GPS devices
  and most mapping APIs return [latitude, longitude]. Shapely follows the GeoJSON
  convention.
  SOLUTION: In check_within_polygon(), Point is constructed as Point(lng, lat) —
  longitude first — to match GeoJSON convention. This is documented in the code
  and validated in unit tests.

CHALLENGE 5 — MinIO Connectivity at Container Start
  MinIO takes a few seconds to become ready after its container starts. FastAPI's
  lifespan function attempting to connect immediately caused intermittent startup
  failures.
  SOLUTION: MinIO initialisation is wrapped in a try/except block that logs a
  warning instead of crashing the server. The application remains functional;
  MinIO-dependent features (photo upload, report download) fail gracefully until
  MinIO is ready.

5.1 SYSTEM PERFORMANCE EVALUATION
------------------------------------

The system was evaluated under controlled load testing conditions using Locust
with 100 simulated concurrent users over a 30-minute period. The test included
a realistic mix of: 60% check-in/check-out operations, 25% history/analytics reads,
10% approval workflow interactions, and 5% WebSocket connections.

LATENCY RESULTS (all values in milliseconds):

| Endpoint                       | P50   | P95   | P99   |
|--------------------------------|-------|-------|-------|
| POST /attendance/checkin       | 45    | 120   | 310   |
| POST /attendance/checkout      | 38    | 95    | 220   |
| GET /attendance/today          | 12    | 35    | 78    |
| GET /analytics/summary         | 28    | 82    | 195   |
| POST /approvals                | 52    | 140   | 330   |
| WebSocket message delivery     | 8     | 22    | 55    |

The check-in endpoint has higher latency than read operations due to the fraud
detection pipeline (external API call to IPQualityScore + geofence computation).
The Redis cache (94% hit rate on geofence data, 87% on IP fraud scores) keeps
this within acceptable bounds for most requests.

THROUGHPUT:
  - Sustained throughput: 850 requests/second across all endpoints
  - Peak throughput: 1,200 requests/second for 60 seconds without errors
  - Error rate under sustained load: 0.03% (HTTP 503 from Nginx rate limiter)

CACHE EFFECTIVENESS:
  Without Redis caching, each check-in would require:
  - 1 database query for site geofence data
  - 1 external API call to IPQualityScore (~200ms avg)
  Total estimated check-in latency without cache: ~450ms P50
  With cache: 45ms P50 — a 10x improvement.

DATABASE QUERY PERFORMANCE:
  - Index on (user_id, created_at) in attendance_records reduces time-range
    queries from O(n) full scan to O(log n) index scan.
  - Geofence polygon stored as text (GeoJSON) and evaluated by Shapely in Python —
    this trades native PostGIS geometry query speed for deployment simplicity
    (no need for PostGIS geometry type columns in application-layer evaluation).

5.2 FRAUD DETECTION EFFECTIVENESS
------------------------------------

The fraud detection system was validated against known attack vectors:

SCENARIO 1 — VPN Check-In Attempt:
  Test: Employee connected via commercial VPN, checked in normally.
  Result: fraud_flags = {"vpn_detected": true}, fraud_score = 0.40
  If score exceeds threshold (0.6), marked invalid; at 0.40, flagged for review.
  Admin sees the record in the anomaly feed with VPN flag highlighted.

SCENARIO 2 — Mock GPS Attack:
  Test: Android developer mode "mock location" feature enabled; fake coordinates
        sent from a home location to a distant office geofence.
  Result: The mock location flag set by the client was recorded; geofence check
          would also reject if coordinates are outside the real geofence.
          fraud_flags = {"mock_gps_reported": true}, fraud_score = 0.35

SCENARIO 3 — Impossible Travel:
  Test: Two check-ins submitted from locations 2,500km apart within 3 minutes.
  Result: Computed speed = 50,000 km/h >> 900 km/h threshold.
          fraud_flags = {"impossible_travel": true}, fraud_score = 0.50
  Combined with other factors, composite score easily exceeds 0.6.

SCENARIO 4 — Buddy Punching (Device Mismatch):
  Test: Employee A's credentials used from Employee B's device (different
        fingerprint hash).
  Result: fraud_flags = {"device_changed": true}, fraud_score = 0.15
  Low individual score but recorded; repeated device changes raise overall risk.

5.5 DISCUSSION
---------------

STRENGTHS OF THE SYSTEM

The composite fraud detection pipeline is the most technically novel aspect of this
system. No surveyed open-source attendance solution combines VPN detection, mock GPS
detection, impossible travel, and device fingerprinting into a single scored
pipeline. The 0.0–1.0 fraud score is more nuanced than a binary flag — it allows
administrators to set organisation-specific thresholds and review borderline cases
rather than blindly accepting or rejecting all records above/below a threshold.

The real-time architecture (Redis pub/sub → WebSocket) provides genuine zero-latency
admin visibility without polling overhead. During testing, a check-in event from an
employee's phone appeared on the admin dashboard within 15–25ms of being processed
by the backend — substantially faster than any polling-based alternative.

The containerised deployment (Docker Compose, single .env file) significantly lowers
the operational barrier compared to manual deployment. An administrator with basic
Docker knowledge can deploy the entire platform — API server, database, cache,
object storage, background workers, proxy, and development email — with three commands.

The self-hostable model addresses a real pain point for organisations with data
sovereignty requirements (healthcare, finance, government) who cannot store employee
location data in third-party SaaS platforms.

LIMITATIONS AND AREAS FOR IMPROVEMENT

1. GPS Indoor Accuracy: The Achilles' heel of any GPS-based attendance system.
   The current mitigation (configurable radius, accuracy indicator) is pragmatic
   but not a complete solution. True indoor positioning would require additional
   hardware (Wi-Fi RSSI positioning, BLE beacons).

2. Client-Side Mock GPS Trust: The mock GPS flag is set by the mobile app — a
   determined attacker who modifies the app binary or submits a direct API call
   can omit this flag. Server-side mock GPS detection (via GPS metadata analysis)
   would require integration with additional detection services.

3. No Payroll Integration: The system deliberately stops at attendance records.
   Integration with payroll systems (Zoho Payroll, ADP, custom) requires additional
   connector development. The REST API provides all necessary data for external
   integration.

4. Single-Region Deployment: The current Docker Compose setup is designed for a
   single server. Multi-region deployment for global organisations would require
   migration to Kubernetes with PostGIS read replicas and Redis Cluster.

5. Report Generation is Synchronous in Demo: PDF generation via Celery is
   implemented but the PDF rendering library (WeasyPrint or ReportLab) adds
   significant container image size. In the current reference deployment, reports
   are generated as CSV with PDF as a configurable option.

CONCLUSION

This project successfully designed and implemented a production-grade geo-location
based attendance system that addresses the core failures of traditional attendance
mechanisms. The key contributions are:

1. A dual-mode geofence engine that supports both circular radius and arbitrary
   polygon boundaries, enabling accurate presence validation across a wide variety
   of physical workplace configurations.

2. A multi-layered fraud detection pipeline that combines network-level signals
   (VPN/proxy detection), device-level signals (mock GPS, fingerprinting), and
   behavioural signals (impossible travel) into a single composite fraud score,
   providing a richer and more actionable fraud assessment than any single check.

3. A real-time operational picture for administrators delivered through WebSocket
   push, eliminating the refresh-and-wait pattern of legacy dashboards and giving
   supervisors true situational awareness.

4. A fully containerised, self-hostable platform that enables organisations with
   data sovereignty requirements to deploy the system on their own infrastructure
   without vendor lock-in.

5. A gamification layer (streak counting, milestone badges) that incentivises
   consistent attendance without management intervention.

The system demonstrates that a small, focused team — leveraging modern open-source
frameworks (FastAPI, React Native, PostgreSQL/PostGIS, Redis, Celery) — can build
a feature-complete, production-ready attendance platform that competes favourably
with commercial SaaS solutions on both features and security posture, while
remaining fully open and extensible.

FUTURE WORK

The following enhancements are identified as high-value extensions for future
development cycles:

1. INDOOR POSITIONING INTEGRATION
   Integrate Wi-Fi RSSI positioning (using Apple WPS or Android WifiManager) and/or
   Bluetooth Low Energy (BLE) iBeacon scanning as a complementary location signal
   for indoor environments where GPS accuracy is insufficient. This would eliminate
   the primary remaining limitation of the current system.

2. FACE RECOGNITION LIVENESS CHECK
   Add an optional face recognition step at check-in — the employee takes a selfie,
   which is compared against their stored reference photo using a liveness detection
   model (to prevent photo spoofing). This would directly address buddy punching
   at the device level. Candidate models: DeepFace, FaceNet, or a cloud API
   (AWS Rekognition, Azure Face API).

3. OFFLINE CHECK-IN WITH CONFLICT RESOLUTION
   In areas with poor network connectivity (underground facilities, rural sites),
   the mobile app should queue check-in events locally and sync when connectivity
   is restored. Conflict resolution logic would be needed to handle events that
   arrive out of order relative to the server's timeline.

4. KUBERNETES DEPLOYMENT AND HORIZONTAL SCALING
   Migrate from Docker Compose to a Kubernetes Helm chart with:
   - PostgreSQL HA (Patroni or CrunchyData PGO)
   - Redis Cluster for horizontal scaling
   - Celery workers autoscaled via KEDA (Kubernetes Event-Driven Autoscaling)
   - Horizontal Pod Autoscaling for FastAPI based on Prometheus CPU/RPS metrics

5. PAYROLL INTEGRATION CONNECTORS
   Build webhook-based connectors for popular payroll systems (Zoho Payroll, ADP,
   Rippling). The attendance data (check-in/out times, late flags, approved manual
   records) would be pushed to the payroll system on a daily sync schedule.

6. SHIFT RECOMMENDATION ENGINE
   Use historical attendance patterns and machine learning (specifically time-series
   clustering) to recommend shift assignments — identifying employees whose natural
   working patterns align with specific shift timings, reducing chronic lateness.

7. ADVANCED ANALYTICS — PREDICTIVE ABSENTEEISM
   Train a gradient boosting model (XGBoost/LightGBM) on historical attendance
   patterns to predict absenteeism risk per employee per day. Alert managers
   proactively when high-risk days are detected, enabling preventative intervention.

8. MULTI-FACTOR AUTHENTICATION (MFA)
   Add TOTP-based MFA (Google Authenticator compatible) for administrator accounts.
   Employee accounts can optionally enable biometric authentication (Face ID /
   fingerprint) via the device's secure enclave as a second factor.

9. PUBLIC API AND WEBHOOK SYSTEM
   Expose a documented public API with API key authentication (separate from
   employee JWT) for third-party integrations. Implement an outbound webhook system
   so external systems can subscribe to attendance events without polling.

10. WHITE-LABEL / MULTI-TENANT SaaS MODE
    Package the platform as a white-label SaaS product: custom branding per
    organisation, subdomain routing (org.platform.com), and Stripe-based subscription
    billing. This would enable the self-hosted platform to be offered as a managed
    service.
